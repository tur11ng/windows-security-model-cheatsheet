Interactive Logon Flow
1 · User presses Ctrl+Alt+Del (SAS captured by Winlogon)
↓
2 · Winlogon collects credentials via Credential Provider
↓
3 · Winlogon calls LsaLogonUser() in LSASS
↓
4 · LSA routes to auth package (MSV1_0 local / Kerberos domain)
↓
5 · Auth package validates credentials (SAM or DC)
↓
6 · LSA creates logon session + access token (NtCreateToken)
↓
7 · Winlogon creates desktop and spawns userinit.exe with the token
SAM Database Internals
  • LocationHKLM\SAM — mapped to %WINDIR%\System32\config\SAM. Locked during runtime. Read requires SYSTEM or backup privilege.
  • Password StorageNT hash = MD4(UTF-16LE(password)). No per-hash salt. Historically protected with SysKey; modernized in Win10 v1709+.
  • RID Allocation500=Administrator, 501=Guest. New local accounts start at 1000.
  • Remote AccessSAM accessible remotely via MS-SAMR (named pipe over SMB). Restricted by default since Win10 1607.
LSASS Credential Cache
  • What's cachedInteractive logons store: NT hash, LM hash if enabled, Kerberos tickets (TGT + service tickets), DES/AES keys derived from password.
  • WDigestLegacy protocol that stored cleartext password reversibly in LSASS. Disabled by default since Win8.1.
  • Credential GuardVSM (Virtual Secure Mode) isolates LSA secrets in a Hyper-V container (LSAIso). Kerberos keys and NT hashes protected from userspace.
  • Protected Process LightLSASS can run as PPL — even processes with SeDebugPrivilege cannot open it for injection without a signed driver.
