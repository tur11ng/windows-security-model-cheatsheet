07 — Service Accounts & User Rights Assignment

URA
User Rights Assignment UserRights CSP Local security policy LSA user-right API Local accounts gMSA Windows LAPS

Windows defines three categories of accounts relevant to authorization. Service accounts are particularly important because they back nearly every system process and are not visible in normal account enumeration tools (net user, etc.) — their authority comes from User Rights Assignment security policy rather than group membership.

How URA Enters the System
  • ConfigurationDefined locally in secpol.msc, through Group Policy, or through MDM Policy CSP. The setting is machine-scoped: it applies to the computer where the policy is effective.
  • StorageThe effective assignment is stored in the Local Security Authority policy database. Domain GPO can overwrite local assignment when policy refresh applies.
  • Names to SIDsLSA resolves users and groups to SIDs. Microsoft recommends SID-form values for policy tooling because account names can be localized or renamed.
  • Evaluation TimeLSASS evaluates logon rights at logon, service start, scheduled task launch, RDP sign-in, and network authentication. Privilege assignments become part of the resulting token.
How Other Subsystems Use It
  • LSASSChecks allow/deny logon rights before creating a logon session. Deny rights override allow rights when both apply through direct or group membership.
  • SCMWhen a service runs under a named account, the Service Control Manager needs that account to have SeServiceLogonRight. Built-in service identities have service-logon behavior by design.
  • Task SchedulerTasks running as a stored account depend on batch-style logon semantics, commonly tied to SeBatchLogonRight and its deny counterpart.
  • RDP / SMB / RPCRemote Desktop depends on remote interactive logon rights; network access paths depend on network logon rights before object ACLs are even considered.
  • SRMAfter logon succeeds, the Security Reference Monitor consumes token privileges such as SeBackupPrivilege, SeDebugPrivilege, and SeTakeOwnershipPrivilege during access checks.
Account Categories
  • UsersStandard low-privilege accounts. Permissions limited to their own profile and explicitly granted resources.
  • AdministratorsMembers of the local Administrators group. Hold Full Control over most computer objects (files, services, registry).
  • Service AccountsBuilt-in pseudo-identities used by SCM. Not enumerable via net user — privileges derive from URA rather than group membership.
Built-in Service Identities
  • LocalService SID S-1-5-19. Minimum privileges on the local machine. On the network, authenticates as anonymous — cannot access remote resources requiring auth.
  • NetworkService SID S-1-5-20. Minimum privileges locally, but on the network authenticates as the computer account — uses the machine's credentials for outbound connections.
  • LocalSystem SID S-1-5-18 (also called SYSTEM). Full Control over most system objects, equivalent to or stronger than Administrators. On the network, authenticates as the computer account. Highest level of access on Windows.
  • Virtual Accounts Per-service automatic accounts (NT SERVICE\<ServiceName>). Same network identity as NetworkService but isolated locally.
  • MSA / gMSA Managed Service Accounts (per-machine) and group Managed Service Accounts (multi-host). Passwords automatically rotated by AD; never known to administrators.
URA Categories

User Rights Assignments split into two categories:

  • Logon Rights Control how and from where an account may authenticate. Examples: SeInteractiveLogonRight, SeNetworkLogonRight, SeBatchLogonRight, SeServiceLogonRight, SeRemoteInteractiveLogonRight. Also includes the corresponding Deny rights which take precedence.
  • User Rights (Privileges) Control access to objects and can override explicit DACL permissions. The Se*Privilege list in Section 06 enumerates these. Built-in service accounts inherit their effective authority from URA, not group membership.
Configuration Surface
URA is configured via Group Policy (Computer Configuration → Windows Settings → Security Settings → Local Policies → User Rights Assignment), local security policy, or MDM UserRights CSP. Treat these settings as full-replacement lists: policy deployment should include every principal that must retain the right, not only the new account being added.

Tiered Administration Model
Microsoft's recommended hardening model uses Deny logon rights to enforce tiering: Tier 0 (DCs, identity systems) accounts denied logon to Tier 1 (servers) and Tier 2 (workstations); Tier 1 denied logon to Tier 2; etc. Prevents credential exposure on lower-tier hosts.
