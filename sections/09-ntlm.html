09 — NTLM Network Authentication

NTLM
Microsoft NTLM NTLM overview

NTLM is a challenge-response protocol used when Kerberos is unavailable (workgroup, non-SPN targets, IP-based connections). The NT hash is derived from the user's password and never leaves the client — instead, a hash-based response to a server challenge is sent.

NTLM 3-Way Handshake
1 · NEGOTIATE (Client → Server): capabilities, domain, workstation, NTLM flags
↓
2 · CHALLENGE (Server → Client): 8-byte server nonce, negotiated flags, target info
↓
3 · AUTHENTICATE (Client → Server): NTLMv2 Response, username, domain, workstation, session key

NTLMv1 vs NTLMv2

  • NTLMv1Response = DES(NT hash, ServerChallenge). No client nonce. Deprecated.
  • NTLMv2Response = HMAC-MD5(NT hash, HMAC-MD5(NT hash, ServerChallenge + ClientChallenge + Timestamp + TargetInfo)).
Channel & Signing Protections
  • SMB SigningEach SMB packet is HMAC-signed with the session key.
  • Channel BindingBinds NTLM auth to a specific TLS channel via channel binding token (CBT).
  • EPAExtended Protection for Authentication — combines SPN binding and channel binding for HTTPS services.
  • LDAP SigningRequires signed LDAP bind. Mandatory for sensitive AD writes.
  • LmCompatibilityLevelGPO setting. Level 5 = NTLMv2 only, refuse LM and NTLMv1 (recommended).
Session Keys
NTLM derives a session key from the NT hash and challenge material. The session key signs and seals subsequent traffic — without it, an intermediary cannot forge or read protected SMB/RPC messages.
