Foundations — The Windows Security Model

How the kernel enforces identity and access. Pure engineering reference: the components that exist, the data structures they manipulate, and the rules they follow when an access decision is made.

03 — Security Access Tokens

TOKEN
Access tokens Token object rights Client impersonation Restricted tokens Mandatory Integrity Control UAC

An access token is the runtime security context that the Security Reference Monitor (SRM) evaluates. It is not just "the user": it is a compact kernel object containing identity, group memberships, privileges, integrity, default security settings, sandbox state, and a link back to the LSASS logon session. The clean mental model is: account data becomes a logon session; the logon session produces one or more tokens; the effective token is consumed by SRM during each access check.

Token Lifecycle

Lifecycle Diagram: From Credentials to a Handle

The lifecycle matters because token "types" are not isolated objects. UAC linked tokens, restricted tokens, AppContainer tokens, and delegation tokens are all variations created at different points in the same pipeline.

StageSubsystemWhat actually happensWhy it matters
1 · Entry pathWinlogon, SCM, Task Scheduler, RDP, SMB, API callerA component requests a logon type: interactive, service, batch, network, unlock, cached interactive, NewCredentials, and so on.The logon type controls which User Rights Assignment entries apply and whether a reusable local token, a network-only token, or a service token is expected.
2 · AuthenticationLSASS + SAM / AD / Kerberos / NTLMLSASS asks an authentication package to prove the account. The result is user SID, group expansion, account flags, claims/device data where applicable, and authentication metadata.This is where identity is proven, but not yet where file or registry access is decided.
3 · Authorization policyLSASS + LSA policyDeny logon rights, allow logon rights, privileges, group attributes, default owner, default DACL, mandatory policy, and UAC filtering rules are composed.User Rights Assignment gates whether a token can exist for that logon path; privileges and SID attributes shape what the token can later do.

Keep These Separate
Account is persistent directory or SAM data. Logon session is LSASS runtime state. Token is the kernel authorization object. Handle is the result of one successful access check.
Shared Token Anatomy
  • User SIDThe account identity. This SID usually becomes the default owner for new objects and can match allow or deny ACEs.
  • Group SIDsNormal, domain, local, logon, resource, capability, and integrity SIDs. Attributes decide whether each SID can allow access, only deny access, or participate in mandatory policy.
  • PrivilegesSystem powers such as SeDebugPrivilege, SeBackupPrivilege, SeImpersonatePrivilege, and SeChangeNotifyPrivilege. A privilege can be present but disabled until a process enables it.
  • Integrity SIDThe token's mandatory integrity level. It is checked before ordinary DACL walking and can block write/read/execute-up operations.
  • Default owner / DACLApplied when the token creates a securable object without an explicit security descriptor.
  • Token source + LUIDsToken source, authentication ID, modified ID, and logon LUID let Windows connect audit events, logon sessions, and duplicated tokens.
  • Optional sandbox fieldsRestricted SIDs, AppContainer SID, capability SIDs, UIAccess, virtualization state, lowbox state, and trust/protection flags.
How SRM Chooses the Effective Token
  • Thread token presentSRM uses the thread impersonation token. The process primary token is temporarily bypassed for that access check.
  • No thread tokenSRM uses the process primary token. This is the normal case for desktop apps, services, and child processes.
  • SID attributesEnabled SIDs can allow or deny. Deny-only SIDs can deny but cannot grant. Disabled SIDs are ignored for normal DACL decisions.
  • Privilege shortcutSome operations are authorized through privileges instead of ordinary ACEs, or require a privilege in addition to object access.
  • Mandatory policyIntegrity and sandbox policy can remove access that the DACL would otherwise grant. They are restrictive layers, not grants.
  • Granted handleAfter the check, code receives a handle with a fixed granted-access mask. Later token changes do not magically increase that existing handle.

How Token Types Link Together

Relationship Map: One Base Model, Several Modifiers

The most important idea is that Windows does not have eight unrelated token systems. It has one token object model, then several ways to attach it, derive it, filter it, label it, or connect it to network credentials.

RelationshipHow to think about itAccess-check consequence
Base → PrimaryA logon creates a token that can be attached to a process. That process token is the normal security identity for code execution.Used whenever the current thread is not impersonating.
Primary → ImpersonationA server keeps its own process token, then one thread temporarily borrows a client token.The thread token overrides the process token for local access checks; delegation level decides whether it can flow remotely.

Why This Map Matters for Security
Most privilege escalation paths are tree-climbing: start with a restricted or low-integrity token and find a primitive that creates or borrows a higher one. SeImpersonatePrivilege is dangerous because any high-privilege client that connects to your process can be impersonated locally — this is the basis for potato-style attacks. UAC bypass exploits trigger the high-integrity linked token without prompting the user. Sandbox escape (AppContainer, restricted token) means reaching the less-restricted parent token. MIC and DACL are independent gates — a DACL Allow ACE does not override an integrity deny; you must clear both.

Token Categories

Read the following categories as roles a token can play. A token can be a primary token and also be filtered by UAC, restricted, low integrity, or AppContainer. Likewise, an impersonation token can carry an integrity level and can be delegation-capable. The labels overlap because they describe different dimensions: attachment, derivation, restriction, trust level, and network behavior.

Primary Tokens

A primary token is the security context attached to a process. It answers the question "who is this process when none of its threads are impersonating?" Explorer, services, scheduled tasks, command shells, and child processes all run with a primary token.

AspectDetailed explanationConnection to other token types
CreationUsually created by LSASS after a successful logon and handed to Winlogon, SCM, Task Scheduler, RDP, or another trusted caller. APIs such as CreateProcessAsUser, CreateProcessWithTokenW, and service startup paths can place a chosen primary token on a new process.UAC, restricted, AppContainer, and network-only tokens can all be primary tokens when they are assigned to a process.
Default useEvery thread in the process uses the primary token unless a thread token is installed. The token supplies SIDs, privileges, integrity, default DACL, audit identity, and network credential linkage.An impersonation token temporarily overrides this default at the thread level.
Process creationCreating a process with a primary token requires the token to be assignable as primary and the caller to have the right token object access. The new process receives a fresh handle table, but its security context starts from that token.A duplicated impersonation token often must be converted or duplicated into a primary token before it can launch a process.
LifetimeThe primary token remains alive while the process or duplicated token handles reference it. The logon session can outlive individual processes when other tokens or credentials still reference it.Group membership and privilege changes normally require a new logon to appear in a new primary token.
Security trapA process may hold powerful privileges even when they are disabled. Code can enable them for a sensitive operation if the privilege is present in the token.Restricted and UAC-filtered primary tokens reduce this ambient authority by removing or disabling SIDs and privileges.
Connection Model
Treat the primary token as the root runtime identity of a process. Other token concepts either override it for a thread, derive a reduced version from it, attach extra sandbox identity to it, or change its trust/network behavior.
Impersonation Tokens

An impersonation token is a thread-level identity override. It lets a server process keep its own primary token while one thread temporarily performs an access check as a client. This is essential for RPC, COM, named pipes, SMB server code, IIS-style worker flows, and many brokered service designs.

Impersonation levelMeaningPractical consequence
AnonymousThe server cannot obtain a usable client identity.Authorization sees an anonymous-style context, so most protected local resources fail.
IdentificationThe server can inspect the caller identity and groups but cannot access resources as that caller.Good for "should I serve this caller?" decisions, not for opening files or registry keys on the caller's behalf.
ImpersonationThe server can use the client identity for local access checks on the same machine.Common local service model: accept a client request, impersonate, open the local object, then revert.
DelegationThe server can carry the client identity to a remote machine.Requires delegation-capable authentication and policy; this is where Kerberos delegation becomes relevant.
  • Where it attachesOnly to a thread. Other threads in the same process keep using the process primary token unless they impersonate too.
  • How it startsCreated by authentication packages, RPC/COM/pipe impersonation calls, SSPI flows, or token duplication. Servers must carefully call revert routines such as RevertToSelf after the protected operation.
  • Privilege sensitivitySeImpersonatePrivilege is powerful because a service that can impersonate a privileged client may be able to use that client's local authority.
  • Primary-token linkSRM chooses the impersonation token first. If no thread token exists, SRM falls back to the process primary token.
Connection Model
Impersonation is not "logging the whole process in as someone else." It is a temporary thread override. A service can be SYSTEM as a process, impersonate Alice on one thread to open Alice's file, then revert and continue running as SYSTEM.
UAC Linked Tokens

A UAC linked-token pair is created for many interactive administrator logons. Windows creates a normal working token that behaves like a standard user most of the time, plus a related elevated token that can be used after consent or credential approval.

  • Filtered tokenThe default desktop token for an administrator under Admin Approval Mode. It is normally Medium integrity. Administrator SIDs remain present but are marked deny-only, and many administrative privileges are removed or disabled. Explorer and normal child processes inherit this token.
  • Full tokenThe elevated partner token. It is normally High integrity, has admin SIDs enabled, and has the administrative privilege set available. Elevated processes are separate processes that use this full token as their primary token.
  • Linked-token pointerThe relationship is stored as token state. Windows can locate the elevated partner when an elevation broker, manifest, shell verb, installer detection, or explicit request triggers UAC.
  • DACL effectBecause admin SIDs are deny-only in the filtered token, they can still match deny ACEs but cannot satisfy admin allow ACEs. This is why a local admin can be denied access until running elevated.
  • Integrity effectThe filtered token usually cannot write to High-integrity objects because MIC blocks write-up before the DACL can grant access. Elevation changes both SID usability and integrity.
  • Standard user distinctionA standard user normally has no linked admin token. An over-the-shoulder elevation prompt creates a process with another account's token, not a linked full token for the standard user's own account.
Connection Model
UAC linked tokens are two primary tokens for the same admin logon session. The filtered token is what normal processes inherit; the full token is selected only for elevated processes. This is why elevation creates a new process context instead of "turning on admin" inside every existing process.
Virtualization and UIAccess Token State

Virtualization and UIAccess are not separate users and usually not separate token objects. They are token state that changes how a process interacts with protected file, registry, and desktop UI resources.

Virtualization allowedWindows marks some medium-integrity legacy desktop processes as eligible for UAC file/registry virtualization. Eligibility depends on factors such as bitness, manifest, elevation, service status, and AppContainer state.Set during token/process creation after UAC policy and application compatibility decisions are known.

The app thinks the write succeeded in the old location.It is a compatibility shim for non-elevated legacy apps, not an authorization grant. The real protected object is still protected.

Connection Model
Virtualization/UIAccess sit beside UAC and integrity, not above them. They modify selected desktop behaviors after Windows already knows the token's user, elevation state, integrity, and application compatibility profile.
Restricted and Write-Restricted Tokens

A restricted token is a deliberately reduced version of another token. It is useful when code should keep the same broad user identity for auditing and profile access, but should lose dangerous authority for the operation it is about to perform.

  • ConstructionWindows duplicates an existing token, then removes privileges, disables groups, marks SIDs deny-only, adds restricting SIDs, or combines those changes. The result can be used as a primary token for a sandboxed process or as an impersonation token.
  • Deny-only groupsA deny-only SID can still trigger deny ACEs, but it cannot satisfy allow ACEs. This preserves negative security boundaries while removing positive authority.
  • Restricting SID listFor normal restricted tokens, access must pass the ordinary DACL evaluation and also be allowed by the restricting SID list. Think of it as an intersection: normal identity AND sandbox allow-list.
  • Write-restricted modeA write-restricted token applies the extra restricting-SID intersection primarily to write-style access. This lets reads stay closer to the original user while writes require explicit sandbox permission.
  • Relationship to UACUAC filtering also removes authority, but it is a specialized admin split-token design. Restricted tokens are a general-purpose API model for lowering authority.
  • Relationship to AppContainerAppContainer builds a stronger package/capability sandbox on top of constrained-token ideas. A plain restricted token has no package identity or capability SID model by itself.
Connection Model
A restricted token starts from another token and subtracts authority. It can still be attached as a primary token to a sandboxed process or as an impersonation token to a thread, so "restricted" describes derivation, not attachment.
AppContainer / Lowbox Tokens

An AppContainer token, historically called a lowbox token, is the token model behind modern Windows app sandboxing. It is not simply "a low-integrity user"; it adds an application/container identity and explicit capability SIDs to a constrained token.

ComponentWhat it meansHow it links to access checks
AppContainer SIDThe package or container identity. It distinguishes one sandboxed app/container from another even when they run under the same user account.Resources can grant directly to this SID when only that package/container should access them.
Capability SIDsPermission-like SIDs for declared abilities such as network, camera, microphone, enterprise resources, or library access.A DACL can grant access to a capability SID, allowing any package that holds that capability unless additional package-specific ACLs narrow it.
Low integrityMost AppContainer work runs at Low integrity, which blocks writes to ordinary Medium-or-higher objects unless labels, brokered APIs, or explicit grants permit them.MIC blocks first, then DACL/capability logic still has to permit the access.
Brokered accessMany sensitive resources are reached through brokers rather than direct object access. The broker runs with more authority and validates app identity/capabilities.The app token proves package identity and capabilities to the broker; the broker performs the privileged operation.
LPACLess Privileged AppContainer reduces ambient compatibility access further, often requiring resources to allow restricted application package identities explicitly.Useful for tighter sandboxes where "all app packages" style grants are too broad.
Connection Model
AppContainer combines several dimensions at once: it is usually a primary token for a sandboxed process, usually Low integrity, derived from the user, restricted in ambient access, and extended with package/capability SIDs. That combination is why it behaves stricter than a plain restricted token.
Mandatory Integrity Token State

Integrity is stored in the token as a mandatory label SID. It is not a group that grants permission. It is a restrictive trust level used by Mandatory Integrity Control before normal DACL evaluation.

  • No write upThe default MIC rule blocks a lower-integrity token from writing to a higher-integrity object, even if the DACL would allow the user SID.
  • Object labelThe object's mandatory label lives in the SACL. Policy bits can express no-write-up, no-read-up, and no-execute-up behavior.
  • UAC linkFiltered admin tokens usually run at Medium; elevated admin tokens usually run at High. The integrity change is one reason elevation changes behavior.
  • Sandbox linkLow integrity is common for AppContainer, browser renderer, and document sandbox designs. It limits writes even when the user account is otherwise allowed.

Connection Model
Integrity is a layer on every practical token category. Primary, impersonation, UAC filtered, elevated, restricted, and AppContainer tokens all carry an integrity level; SRM checks that level before ordinary allow ACEs can help.
Network-Only and Delegation Tokens

Network-related token behavior is where local and remote identity can diverge. A process can look like one user locally while using another credential set for outbound authentication, or a server can be allowed to carry a client's identity to another machine.

  • Network-only / NewCredentialsUsed by flows such as runas /netonly and LOGON32_LOGON_NEW_CREDENTIALS. Locally, the token mostly behaves like the caller's existing identity. For outbound SSPI authentication, supplied network credentials can be used.
  • Validation timingThe local machine can create a NewCredentials logon session without proving the supplied remote password to a domain controller immediately. The remote service validates the credentials when they are actually used.
  • Local vs remote accessOpening a local file, registry key, or process uses the local token identity. Accessing SMB, LDAP, HTTP, SQL, or another remote service can present the outbound credentials associated with the logon session.
  • Delegation tokenAn impersonation token at Delegation level lets a server access a remote resource as the client. In domain environments this normally depends on Kerberos delegation settings and account restrictions.
  • Credential protection linkRestricted Admin, Remote Credential Guard, Protected Users, and "account is sensitive and cannot be delegated" style controls change whether reusable credentials or delegation-capable tickets are available.
  • Impersonation linkIdentification and ordinary impersonation are local concepts. Delegation is the level that crosses machine boundaries, so it is the bridge between token mechanics and Kerberos delegation policy.
Connection Model
Network behavior hangs from the logon session and the impersonation level. NewCredentials changes outbound authentication while local access still uses the local token. Delegation changes whether a client's impersonation token can be used beyond the current machine.

04 — Security Descriptors

SD
Security descriptors ACLs and ACEs SDDL format ACE strings

Every securable kernel object carries a security descriptor. The SRM reads it during every access check. A descriptor can be in absolute form (pointers to separate ACL buffers) or relative form (self-contained flat buffer — used for persistence in registry/file).

Security Descriptor Components
  • Owner SIDIdentity that owns the object. Owner automatically gets ReadControl + WriteDac regardless of DACL.
  • Primary Group SIDPOSIX compatibility field. Rarely enforced on Windows but present.
  • DACLDiscretionary ACL — controls who can access the object. A NULL DACL grants everyone full access. An empty DACL denies all access.
  • SACLSystem ACL — controls auditing and integrity labels. Requires SeSecurityPrivilege to read/write.
  • Control FlagsBitfield: DaclPresent, SaclPresent, DaclProtected, SaclProtected, DaclAutoInherited, etc.
Critical Distinction
A NULL DACL (absent) = unrestricted access to everyone. An empty DACL (present but zero ACEs) = no access to anyone. Missing DACL ≠ secure — it is maximally permissive.

An ACE (Access Control Entry) is one rule in an ACL.

ACE Ordering (Mandatory)

The OS enforces strict ordering when building a DACL. Evaluation stops at first full match.

1 · Explicit Deny ACEs (non-inherited)
2 · Explicit Allow ACEs (non-inherited)
3 · Inherited Deny ACEs
4 · Inherited Allow ACEs

Explicit ACEs always take precedence over inherited ones. Within explicit ACEs, Deny beats Allow.

Access Rights Taxonomy
Generic (GRFA: GenericRead/Write/Execute/All) — mapped to specific rights via GenericMapping.
Standard — Delete, ReadControl, WriteDac, WriteOwner, Synchronize. Present on all object types.
Specific — Defined per object type (FILE_READ_DATA, KEY_SET_VALUE, etc.).
SDDL — Security Descriptor Definition Language

Text representation of a security descriptor. Used in policies, SCE templates, and AD.

Absolute vs Self-Relative Form

The same descriptor can be represented in two memory layouts. Confusing these is a common source of broken parsers and incorrect assumptions.

FormWhere usedMeaning
AbsoluteRuntime APIsThe descriptor contains pointers to separate owner, group, DACL, and SACL buffers.

Practical Rule
Registry, file system, AD, and security policy often persist descriptors as self-relative binary or SDDL. API code often converts to absolute form before modifying individual parts.
ACL and ACE Binary Shape
  • ACL headerContains ACL revision, total ACL size, and ACE count. ACEs follow immediately after the header.
  • ACE headerEvery ACE begins with type, flags, and size. Unknown ACE types can be skipped using the size field.
  • Access maskThe rights bitmask granted, denied, audited, or filtered by that ACE.
  • SIDThe principal to which the ACE applies. For object ACEs, GUID fields can appear before the SID.
  • Extra dataCallback, resource attribute, scoped policy, and conditional ACEs may carry additional data after the normal fields.
Creation and Inheritance
  • Creator descriptorThe caller can pass a descriptor when creating an object. If omitted, the token default DACL is used.
  • Parent inheritanceContainer objects can contribute inheritable ACEs to children. Files inherit from folders; AD objects inherit from containers/OUs.
  • Merge behaviorWindows merges explicit creator ACEs with parent inheritable ACEs unless inheritance protection flags suppress it.
  • Auto-inheritedInherited ACEs are marked so tools can distinguish explicit permissions from inherited permissions.
Special SIDs in ACEs
  • OWNER RIGHTSLets a descriptor explicitly restrict or define what the owner receives instead of relying only on implicit owner rights.

05 — The Access Check Process

AC
Access control overview DACL evaluation How AccessCheck works AccessCheck AccessCheckByType Generic rights GENERIC_MAPPING

When a thread opens a handle to a kernel object, the SRM runs a deterministic three-phase check. All three phases must pass. The SRM is the only component that performs this — no kernel subsystem bypasses it.

01
Mandatory Integrity Check
Compare the token's Integrity Level against the object's Mandatory Label in the SACL.

No-Write-Up (default): if token IL < object IL → deny write access.
No-Read-Up: optional — blocks reads from lower IL.
No-Execute-Up: optional — blocks execute from lower IL.

This check precedes DACL evaluation — even if the DACL would grant access, a MIC violation blocks it.
02
Token Access Check
Owner Check: if the token's user SID (or any enabled group SID) matches the object's Owner, the caller is automatically granted ReadControl + WriteDac.

Privilege Check: certain privileges grant additional implicit access:
SeBackupPrivilege → read any file regardless of DACL.
SeRestorePrivilege → write any file regardless of DACL.
SeTakeOwnershipPrivilege → WriteOwner on any object.
SeSecurityPrivilege → read/write SACL.
03
Discretionary Access Check
Walk the DACL ACE list in order. For each ACE, check if the ACE's SID is present (enabled) in the token.

Deny ACE match: any requested bit denied → immediately DENY entire request.
Allow ACE match: granted bits accumulate in a running bitmask.
Result: GRANT if all requested bits are in accumulated allows and none were denied.

NULL DACL → GRANT all. Empty DACL → DENY all.
Sandboxed Token Access Checks
  • Restricted TokenTwo parallel DACL walks: one with normal token SIDs, one with the Restricting SIDs list. Access granted only if both walks independently grant access.
  • Lowbox / AppContainerThird additional check: the resource must have an ACE allowing the AppContainer SID and/or the required Capability SID.
  • Enterprise ChecksCentral Access Policies (from AD) can add an additional claim-based check layer using callback ACEs.
Traversal & Other Checks
  • Traversal CheckOpening C:\A\B\C.txt requires Traverse on \A and \A\B. SeChangeNotifyPrivilege (held by all users) bypasses per-directory traversal.
  • Handle DuplicationDuplicating a handle requires access to the source process object. Cannot exceed original rights.
  • Limited ChecksSome operations (e.g., listing directory contents) perform abbreviated checks.
  • Object Type CheckFor AD/some kernel objects, ACEs can be scoped to a specific object type or attribute GUID.
Before the DACL Walk
  • Requested accessThe caller asks for a bitmask such as Read, Write, Delete, WriteDac, GenericAll, or MAXIMUM_ALLOWED.
  • Generic mappingGenericRead/Write/Execute/All are translated into object-specific bits. A file, process, token, registry key, and AD object all map generics differently.
  • Valid access maskBits that are meaningless for that object type are stripped or rejected. Object types define what rights exist.
  • Effective tokenIf a thread token exists, SRM uses it. Otherwise SRM uses the process token.
  • Previously grantedSome rights may already be granted by ownership or privileges before the DACL loop begins.
Remaining Access Algorithm

The DACL walk is easiest to understand as a shrinking set of bits called RemainingAccess.

Remaining = Desired - AlreadyGranted
for each ACE in canonical order:
  if ACE does not apply: continue
  if Deny intersects Remaining: deny
  if Allow intersects Remaining: grant those bits
  if Remaining is empty: grant

Canonical ordering matters because the loop can stop early. If allows appear before denies, a bad DACL may grant everything before the deny ACE is reached.

SID Matching Rules
  • Allow ACEMatches token user SID, enabled group SIDs, enabled capability SIDs, or relevant package SIDs.
  • Deny ACEMatches enabled SIDs and deny-only SIDs. This is why UAC filtered admin SIDs can still be denied.
  • Disabled SIDIgnored for normal allow/deny checks.
  • OWNER RIGHTSIf present, can replace the broad implicit owner behavior with explicit owner rights.
  • SELFDuring object-specific checks, SELF is replaced with the principal SID of the object being checked.
MAXIMUM_ALLOWED
  • PurposeAsk Windows to compute the largest access mask the token can receive instead of requesting exact bits.
  • Deny still appliesA deny ACE removes matching bits from the possible result.
  • Privileges countOwner rights and enabled privileges can add rights to the computed result.
  • Use carefullyOpening handles with broader rights than needed increases damage if the handle is leaked or duplicated.
Privilege Short-Circuits
  • SeBackupPrivilegeMay grant backup/read-style access even when the DACL denies ordinary reads.
  • SeRestorePrivilegeMay grant restore/write-style access and override ownership-related restrictions.
  • SeTakeOwnershipPrivilegeMay grant WriteOwner even when DACL says no.
  • SeSecurityPrivilegeRequired to read or write SACLs through ACCESS_SYSTEM_SECURITY.
  • SeChangeNotifyPrivilegeBypasses directory traversal checks, not ordinary file read/write checks.
Object-Type Access Checks

Some resources, especially AD objects, are not just "one object with one mask." They have properties, property sets, child object classes, validated writes, and extended rights.

  • AllowedObject ACEGrants access only when the requested object type GUID matches the ACE's object type GUID.
  • DeniedObject ACEDenies only the matching property, property set, class, or extended right.
  • InheritedObjectTypeControls which child object classes inherit an ACE.
  • Result listObject-type APIs can return a result per requested object type, not only one yes/no decision.
Central Access Policy Layer
  • Resource attributesClaims stored on the resource, such as classification or department.
  • User/device claimsClaims in the token that describe the authenticated user or device.
  • Scoped policyA SACL ACE points the object to the Central Access Policy that should apply.
  • Callback ACEsConditions are evaluated in addition to normal DACL checks. CAP can remove access that the DACL alone would allow.
End-to-End Access Check Mental Model
StepQuestionIf it fails

07 — Service Accounts & User Rights Assignment

URA
User Rights Assignment UserRights CSP Local security policy LSA user-right API Local accounts gMSA Windows LAPS

Windows defines three categories of accounts relevant to authorization. Service accounts are particularly important because they back nearly every system process and are not visible in normal account enumeration tools (net user, etc.) — their authority comes from User Rights Assignment security policy rather than group membership.

How URA Enters the System
  • ConfigurationDefined locally in secpol.msc, through Group Policy, or through MDM Policy CSP. The setting is machine-scoped: it applies to the computer where the policy is effective.
  • StorageThe effective assignment is stored in the Local Security Authority policy database. Domain GPO can overwrite local assignment when policy refresh applies.
  • Names to SIDsLSA resolves users and groups to SIDs. Microsoft recommends SID-form values for policy tooling because account names can be localized or renamed.
  • Evaluation TimeLSASS evaluates logon rights at logon, service start, scheduled task launch, RDP sign-in, and network authentication. Privilege assignments become part of the resulting token.
How Other Subsystems Use It
  • LSASSChecks allow/deny logon rights before creating a logon session. Deny rights override allow rights when both apply through direct or group membership.
  • SCMWhen a service runs under a named account, the Service Control Manager needs that account to have SeServiceLogonRight. Built-in service identities have service-logon behavior by design.
  • Task SchedulerTasks running as a stored account depend on batch-style logon semantics, commonly tied to SeBatchLogonRight and its deny counterpart.
  • RDP / SMB / RPCRemote Desktop depends on remote interactive logon rights; network access paths depend on network logon rights before object ACLs are even considered.
  • SRMAfter logon succeeds, the Security Reference Monitor consumes token privileges such as SeBackupPrivilege, SeDebugPrivilege, and SeTakeOwnershipPrivilege during access checks.
Account Categories
  • UsersStandard low-privilege accounts. Permissions limited to their own profile and explicitly granted resources.
  • AdministratorsMembers of the local Administrators group. Hold Full Control over most computer objects (files, services, registry).
  • Service AccountsBuilt-in pseudo-identities used by SCM. Not enumerable via net user — privileges derive from URA rather than group membership.
Built-in Service Identities
  • LocalService SID S-1-5-19. Minimum privileges on the local machine. On the network, authenticates as anonymous — cannot access remote resources requiring auth.
  • NetworkService SID S-1-5-20. Minimum privileges locally, but on the network authenticates as the computer account — uses the machine's credentials for outbound connections.
  • LocalSystem SID S-1-5-18 (also called SYSTEM). Full Control over most system objects, equivalent to or stronger than Administrators. On the network, authenticates as the computer account. Highest level of access on Windows.
  • Virtual Accounts Per-service automatic accounts (NT SERVICE\<ServiceName>). Same network identity as NetworkService but isolated locally.
  • MSA / gMSA Managed Service Accounts (per-machine) and group Managed Service Accounts (multi-host). Passwords automatically rotated by AD; never known to administrators.
URA Categories

User Rights Assignments split into two categories:

  • Logon Rights Control how and from where an account may authenticate. Examples: SeInteractiveLogonRight, SeNetworkLogonRight, SeBatchLogonRight, SeServiceLogonRight, SeRemoteInteractiveLogonRight. Also includes the corresponding Deny rights which take precedence.
  • User Rights (Privileges) Control access to objects and can override explicit DACL permissions. The Se*Privilege list in Section 06 enumerates these. Built-in service accounts inherit their effective authority from URA, not group membership.
Configuration Surface
URA is configured via Group Policy (Computer Configuration → Windows Settings → Security Settings → Local Policies → User Rights Assignment), local security policy, or MDM UserRights CSP. Treat these settings as full-replacement lists: policy deployment should include every principal that must retain the right, not only the new account being added.

NTLM is a challenge-response protocol used when Kerberos is unavailable (workgroup, non-SPN targets, IP-based connections). The NT hash is derived from the user's password and never leaves the client — instead, a hash-based response to a server challenge is sent.

Session Keys
NTLM derives a session key from the NT hash and challenge material. The session key signs and seals subsequent traffic — without it, an intermediary cannot forge or read protected SMB/RPC messages.

10 — Kerberos Authentication

KRB5
Kerberos overview

Kerberos v5 is the default protocol for domain authentication. It is ticket-based and mutual — both client and server authenticate each other. The KDC (Key Distribution Center) runs on every Domain Controller. No passwords travel the network; all encryption uses keys derived from account passwords.

Components
  • KDCKey Distribution Center. Runs on DC. Contains AS + TGS + database of all account keys.
  • ASAuthentication Service. Issues TGTs in response to valid pre-auth.
  • TGSTicket Granting Service. Issues service tickets in exchange for valid TGTs.
  • PrincipalAny identity: user, computer, or service account.
  • SPNService Principal Name. Format: service/host@REALM.
  • RealmDomain (uppercase). Cross-realm trusts enable inter-domain auth.
  • krbtgtSpecial account whose hash encrypts ALL TGTs.
Ticket Types & Lifetime
  • TGTTicket Granting Ticket. Issued by AS. Encrypted with krbtgt hash. Default 10 hours, renewable up to 7 days.
  • Service TicketIssued by TGS. Encrypted with target service account's hash.
  • PACPrivileged Attribute Certificate embedded in tickets. Contains user SID, group SIDs, logon info. Signed by KDC.
  • AuthenticatorClient-generated timestamp encrypted with session key. Sent with each ticket use to prevent replay.
Encryption Types
  • RC4-HMACUses NT hash directly as key. Legacy.
  • AES-128/256Uses PBKDF2-derived key from password + salt (username@domain). Modern standard.
  • DES-CBC-CRCLegacy, disabled by default since Win7/2008R2.

Pre-authentication

By default, the AS-REQ includes a timestamp encrypted with the user's long-term key. This proves the client knows the password before the KDC issues a TGT. Without it, the AS-REP becomes offline-crackable.

Delegation allows a service to act on behalf of a user against another service — necessary for multi-tier applications.

  • Constrained (KCD) Front-end service can only delegate to a specific list of SPNs in msDS-AllowedToDelegateTo. Uses S4U2Self (obtain a self-ticket for any user — protocol transition) and S4U2Proxy (use that ticket to get one for the back-end service).

13 — Privilege Escalation (Token & Privilege Abuse)

TA0004
Access tokens Privileges UAC

Privilege escalation [TA0004] is the tactic of going from limited access — typically a standard user shell after initial access — to higher authority on the host (Administrator, SYSTEM) or in the domain. Most local techniques exploit the token model, the privilege list, or UAC. Service-related path/DLL hijacking is covered separately in Section 14.

SeImpersonatePrivilege Abuse — Potato Family T1134.001 / T1134.002

Any service account holding SeImpersonatePrivilege (LocalService, NetworkService, IIS application pool identities, MSSQL service, etc.) can impersonate any security context that authenticates to it. The attacker coerces SYSTEM to authenticate to a local listener under attacker control — historically via DCOM/RPC (RottenPotato, JuicyPotato), the Print Spooler and Print Notification (PrintSpoofer), or COM type-coercion (RoguePotato, GenericPotato) — captures the resulting authentication, and uses the impersonation token to spawn a SYSTEM process.

Co-occurs with SeAssignPrimaryTokenPrivilege, which permits assigning the impersonation token as a primary token of a new process directly.

Mitigations
  • Remove SeImpersonatePrivilege from accounts that don't need it; review URA on web/SQL servers
  • Apply latest patches — Microsoft has progressively closed coercion vectors (Spooler, EFS-RPC, DCOM type confusion)
  • Disable Print Spooler service on hosts that don't need it (servers, DCs)
  • Use AppContainers / virtual service accounts where possible to limit blast radius
  • Monitor 4672 and 4673 events for unusual privilege use, and 4688 for SYSTEM children of non-system parents
SeBackupPrivilege / SeRestorePrivilege Abuse T1003.002

These privileges bypass DACL checks for read/write respectively. SeBackupPrivilege permits opening any file or registry hive — including locked SAM and SECURITY hives — and exporting them for offline analysis. SeRestorePrivilege permits writing arbitrary files, including system binaries that run as SYSTEM at next boot. Backup Operators is the canonical privileged group, but third-party backup software often grants these rights to service accounts.

Mitigations
  • Treat Backup Operators as Tier-0 equivalent; restrict membership and audit rigorously
  • Ensure backup service accounts use gMSAs and are isolated from interactive use
  • Enable Credential Guard so SAM/SECURITY extraction yields encrypted blobs only
  • Monitor 4673 (privileged service called) when these privileges are exercised
SeDebugPrivilege — LSASS Access T1003.001

Permits opening any process for full read/write access regardless of its DACL. The canonical use is opening LSASS to extract credentials from memory. Held by Administrators by default but disabled — must be explicitly enabled in the token. Many anti-malware drivers also rely on this privilege, complicating policy tightening.

Mitigations
  • Enable LSASS as a Protected Process Light (PPL) — blocks userland injection even with SeDebugPrivilege
  • Enable Credential Guard — isolates secrets in VSM (LSAIso) so memory dumps don't contain usable material
  • Restrict SeDebugPrivilege via URA to only the accounts that genuinely need it
  • Deploy EDR with kernel callbacks (PsSetCreateProcessNotifyRoutineEx, ObRegisterCallbacks) to detect LSASS handle requests
  • Remove debug privilege from local Administrators on workstations where developers don't operate
SeLoadDriverPrivilege — Bring Your Own Vulnerable Driver (BYOVD) T1068

Loading a kernel driver yields immediate Ring 0 code execution — the maximum privilege available on the host. Microsoft requires drivers to be signed, but many legitimately signed drivers contain vulnerabilities (arbitrary read/write primitives) that the attacker can chain to execute kernel shellcode. Public lists of known-vulnerable signed drivers exist (loldrivers.io).

Mitigations
  • Enable Microsoft Vulnerable Driver Blocklist (built-in since Win11; configurable in WDAC)
  • Require Hypervisor-Protected Code Integrity (HVCI) — blocks unsigned and disallowed-signature drivers
  • Restrict SeLoadDriverPrivilege via URA
  • Enable Smart App Control / WDAC policies in audit then enforce mode
SeTakeOwnershipPrivilege — DACL Rewrite T1222.001

Permits taking ownership of any object. Once owned, the actor automatically has WriteDac (per Token Access Check Phase 02) and can rewrite the DACL to grant themselves any access — including writing to system binaries or rewriting service registry keys.

Mitigations
  • Audit and minimize URA assignment of this privilege
  • Monitor 4670 (Permissions on object changed) and 4674 (privileged operation attempted)
  • WDAC code integrity policy prevents executing tampered binaries even after takeover
UAC Bypass T1548.002

UAC produces a Medium IL filtered token from an admin's split token. Bypass techniques exploit the auto-elevation behavior of trusted Microsoft binaries that elevate without prompting, plus a writable input that the trusted binary consumes. Common vectors: registry/COM hijacks targeting fodhelper.exe, computerdefaults.exe, eventvwr.exe and similar; environment variable hijacks against sdclt.exe; DLL hijacks against signed elevated binaries with insecure search paths; and ICMLuaUtil / IFileOperation COM elevation moniker abuse. The attacker writes to a per-user registry/key/file consumed by the auto-elevating binary, then triggers it; the resulting child process runs at High IL.

Mitigations
  • Set UAC to Always Notify (highest) — disables silent auto-elevation
  • Remove standard users from the local Administrators group; use LAPS for emergency local admin
  • Apply Secure Desktop for elevation prompts (default; do not disable)
  • Disable EnableLUA=0 alarms in monitoring — disabling UAC is itself an indicator
  • WDAC / AppLocker to constrain which binaries can execute as elevated
Token Impersonation / Theft (Generic) T1134

An attacker with sufficient access to a process holding a privileged token can duplicate its primary or impersonation token and either impersonate the security context (thread-level) or spawn a new process with the duplicated primary token. Common with SYSTEM tokens already present in service processes when the attacker first achieves admin-level local access.

Mitigations
  • Restrict SeImpersonatePrivilege and SeAssignPrimaryTokenPrivilege
  • Run sensitive services as Protected Processes where supported
  • EDR detection on token duplication / DuplicateTokenEx + CreateProcessAsUser sequences
  • Network segmentation so a SYSTEM-on-host token has limited lateral reach
Generic Software Vulnerabilities — Local EoP T1068

Memory corruption or logic bugs in kernel components, IPC endpoints (ALPC, RPC), drivers, or privileged user-mode services that allow a low-privileged process to coerce execution at higher integrity. Frequently chained from sandbox escapes or used directly when a vulnerable third-party driver/service is present. Reverse engineering with Ghidra or IDA against native binaries (and dnSpy/dotPeek for .NET services) is the standard discovery workflow.

Mitigations
  • Aggressive patching cadence — local EoP CVEs are routinely weaponized within days
  • Exploit Protection (DEP, ASLR, CFG, CET, ACG, Code Integrity Guard) for user-mode services
  • Reduce attack surface: disable unused services, restrict third-party kernel components
  • WDAC / HVCI to constrain post-exploitation

14 — Vulnerable Services & Binary Hijacking

SVC
Service accounts Event 4670

Services are one of the largest local attack surfaces because most run as SYSTEM and many are configured with weak file system, registry, or path-resolution properties. Discovery is straightforward via sc query, wmic service, or PowerShell Get-Service / Get-CimInstance Win32_Service; automated tooling such as PowerSploit's PowerUp module and SharpUp enumerate misconfigurations. The categories below all yield SYSTEM-level code execution when the service is privileged.

DLL Search Order Hijacking T1574.001

When a process loads a DLL by name without an absolute path, Windows searches a defined sequence of directories. If the attacker can plant a malicious DLL in a directory that is searched before the legitimate location, the malicious DLL is loaded into the privileged process. Common when a service binary's own application directory or any directory in the PATH is writable by lower-privileged users. The default search order roughly is: KnownDLLs → application directory → System32 → System → Windows → current directory → PATH directories — but this is altered by SafeDllSearchMode, manifests, and side-by-side configuration.

Mitigations
  • Ensure SafeDllSearchMode is enabled (default) so the current directory is searched last
  • Use the KnownDLLs registry list — these are loaded from System32 only and cannot be hijacked
  • Audit service install directories — they should not be writable by non-admins
  • Use absolute paths or LOAD_LIBRARY_SEARCH_* flags in service code
  • WDAC publisher rules block unsigned/untrusted DLLs from loading into protected processes
PATH Environment Variable Hijacking T1574.007

If the system or service PATH contains a directory writable by low-privileged users, the attacker plants an executable matching a name the service or its child processes invoke without an absolute path. The injected binary is found first in PATH lookup and executes at the service's privilege level. Often arises from sloppy installer behavior that prepends application directories with weak ACLs to system PATH.

Mitigations
  • Audit system PATH for entries with weak ACLs (icacls on each directory)
  • Remove writable directories from system PATH; place application binaries in protected directories
  • Always invoke binaries with full paths in scripts and service configurations
Search Order Hijacking (Executable) T1574.008

The executable equivalent of DLL search order hijacking — when a process spawns a child by name without an absolute path, the same directory search rules apply, and a malicious binary higher in the search order will be executed instead.

Mitigations
  • Always specify absolute paths in CreateProcess / ShellExecute service code
  • Sanitize PATH; remove user-writable entries
  • AppLocker / WDAC publisher rules block unauthorized binaries from running
Unquoted Service Path T1574.009

When a service's BINARY_PATH_NAME contains spaces and is not enclosed in quotes (e.g. C:\Program Files\Bad Service\bin\svc.exe), the SCM tries each tokenized prefix as a candidate executable: C:\Program.exe, C:\Program Files\Bad.exe, C:\Program Files\Bad Service\bin\svc.exe. If the attacker can write a binary at one of the earlier candidate paths, the SCM launches it as the service identity (often SYSTEM).

The frequency of this misconfiguration in third-party software, combined with the default writable status of C:\ root for some legacy installers, makes this a perennial finding.

Mitigations
  • Quote service binary paths properly: sc config <svc> binpath= "\"C:\Program Files\...\"
  • Audit ACLs on C:\ and intermediate directories — should not be writable by non-admins
  • Periodic scan via wmic service get name,pathname,startmode piped through unquoted-path detection
Weak Service File Permissions T1574.010

The service's executable, or any DLL it loads, has a DACL granting write access to a low-privileged identity (Users, Authenticated Users, or a specific user). The attacker overwrites or replaces the binary with a malicious payload; on next service start (or after forcing a restart), the SCM executes the attacker-controlled binary as the service principal.

Mitigations
  • Audit DACLs on all service binaries and dependencies — only Administrators / SYSTEM should have Modify
  • WDAC code integrity policy — even if the file is replaced, an unsigned/untrusted replacement won't execute
  • Monitor 4663 events with WriteData / AppendData on service executables
Weak Service Registry Permissions T1574.011

The service's configuration key under HKLM\SYSTEM\CurrentControlSet\Services\<ServiceName> is writable by a low-privileged identity. The attacker rewrites ImagePath to point to their own binary, or modifies ObjectName to change the run-as identity. Next service start uses the new configuration. Even without write access to the binary, controlling the registry key fully owns the service.

Mitigations
  • Audit registry DACLs on all service keys — only SYSTEM/Admins should have SetValue
  • Monitor 4657 (Registry value changed) on service ImagePath / ObjectName attributes
  • Group Policy: Restricted Groups / Security Templates to enforce baseline service ACLs
Weak SCM ACL on Service Object T1574.011

Even when binaries and registry keys are protected, the service object itself — managed by the SCM — has its own DACL. If a low-privileged user holds SERVICE_CHANGE_CONFIG on the service, they can call ChangeServiceConfig to alter the binPath or the run-as account directly through SCM APIs. SERVICE_ALL_ACCESS on a SYSTEM service is equivalent to local SYSTEM compromise.

Mitigations
  • Audit per-service SDDL with sc sdshow <service> — verify only SYSTEM/Admins have CC/CR (CHANGE_CONFIG)
  • Group Policy security templates can enforce service security descriptors
  • Monitor 4670 (object permissions changed) on Service objects

15 — Credential Access

TA0006
Credential Guard DPAPI CryptProtectData DPAPI CryptUnprotectData

Once local admin or SYSTEM is achieved, the next objective is harvesting credentials for lateral movement and persistence. Targets are LSASS memory, the SAM/SECURITY hives, the DPAPI master key chain, and (on a DC) the directory database itself.

LSASS Memory Dumping T1003.001

LSASS holds, for every interactive logon session on the host, the user's NT hash, Kerberos AES/RC4 keys and tickets, and (if WDigest enabled) cleartext password. Dumping the process — via Mimikatz, comsvcs.dll MiniDump, taskmgr, ProcDump, or direct MiniDumpWriteDump — yields all of those secrets for offline parsing. Requires SeDebugPrivilege to open the process.

Mitigations
  • Run LSASS as Protected Process Light — registry RunAsPPL=1 (or =2 on Win11 22H2+ for boot-time enforcement)
  • Enable Credential Guard (VSM) — secrets isolated in LSAIso, dumps yield encrypted material
  • EDR with kernel callbacks to alert on LSASS handle requests with PROCESS_VM_READ
  • Enable Attack Surface Reduction rule "Block credential stealing from LSASS"
  • Restrict SeDebugPrivilege
SAM & SECURITY Hive Extraction T1003.002 / T1003.004

The SAM hive contains local NT hashes; the SECURITY hive contains LSA Secrets (service account passwords stored in cleartext, cached domain credentials as DCC2 verifiers, DPAPI keys). The hives are locked at runtime, so direct file copy fails — but Volume Shadow Copy, reg save with SeBackupPrivilege, or raw NTFS access (ntfsclone, esentutl /y) all bypass the lock. Decryption uses the SYSTEM hive's BootKey (legacy) or DPAPI machine key (modern).

Mitigations
  • Disable cleartext WDigest, restrict use of cached domain credentials (CachedLogonsCount)
  • Use LAPS so even if local Administrator hash is recovered, it cannot be reused
  • Disable VSS where not required; monitor VSS shadow creation events (8222)
  • Monitor 4663/4670 events on \Windows\System32\config\SAM and SECURITY
  • EDR detection on reg save HKLM\SAM and similar patterns
DCSync — Replication of Domain Secrets T1003.006

The MS-DRSR replication protocol allows a DC (or any principal granted the DS-Replication-Get-Changes + DS-Replication-Get-Changes-All extended rights on the domain root) to request replication of secret attributes — including the NT hash and Kerberos keys — for any account in the domain, krbtgt included. The attack runs against any DC over the network, requiring no code execution on the DC. Domain Admins, Enterprise Admins, and Domain Controllers hold these rights by default; misconfigured AD ACLs frequently grant them to other accounts.

Mitigations
  • Audit and minimize accounts holding DS-Replication-Get-Changes-All on the domain root (use the Active Directory module's DSACLs / ACL queries)
  • Enable advanced auditing for Directory Service Access (4662) on the domain object — alert on replication right exercise from non-DC sources
  • Network segment DCs; DC-to-DC replication should not be reachable from user subnets
  • Tier-0 enforcement: identity admins never log into Tier-1/2 systems
DPAPI Master Key Abuse T1555.003 / T1003.004

The Data Protection API encrypts user secrets — saved browser passwords, RDP credentials, certificate private keys — with a per-user master key derived from the user's password (or, for domain users, backed by a domain-wide DPAPI backup key on a DC). With local SYSTEM, an attacker can extract user master keys; with the domain DPAPI backup key (extractable via LSADUMP::backupkeys against a DC), they can decrypt any user's protected blob in the domain offline, even without that user's hash.

Mitigations
  • Treat the DPAPI domain backup key as a Tier-0 secret; restrict access on DCs
  • Credential Guard isolates DPAPI keys within VSM
  • EDR detection on LSARPC LsarRetrievePrivateData calls referencing G$BCKUPKEY*
  • Discourage browser-stored credentials; use centralized password managers with hardware-backed keys
Cached Domain Credentials (DCC2) T1003.005

For offline-logon support, Windows stores up to 10 (default) domain users' password verifiers in HKLM\SECURITY\Cache. The format (DCC2/MSCASHv2) is PBKDF2-HMAC-SHA1 over the username and NT hash and is not directly usable for authentication — but is offline-crackable. A successful crack gives the cleartext password, which can be replayed against any service.

Mitigations
  • Reduce CachedLogonsCount to 0 or 1 on hosts that don't go offline
  • Strong password policies (length >14) — DCC2 cracking depends on password strength
  • Add privileged users to Protected Users group — disables credential caching for those accounts
WDigest Re-enable T1112 + T1003

WDigest stored cleartext passwords reversibly in LSASS. Disabled by default since Win8.1, but the registry value HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest\UseLogonCredential = 1 re-enables it. After flipping the value, the attacker waits for users to authenticate (or forces RDP reconnections) — subsequent LSASS dumps yield cleartext credentials.

Mitigations
  • Group Policy enforcement of UseLogonCredential = 0 with refresh interval
  • Monitor 4657 events on the WDigest registry path
  • Credential Guard prevents extraction even if WDigest re-enabled

16 — NTLM Attacks

NTLM-OFF
Microsoft NTLM NTLM overview

NTLM's challenge-response design is vulnerable to relay (because the client cannot verify the server's identity in many configurations), to offline cracking (because the response is computed deterministically from the NT hash), and to credential reuse (because the NT hash itself is a sufficient authenticator).

NTLM Relay T1557.001

The attacker positions themselves between client and server (LLMNR/NBT-NS/mDNS poisoning, ARP spoofing, malicious WPAD, IPv6 DHCPv6 takeover, or coercion via PrinterBug, PetitPotam, DFSCoerce, Coerced Authentication). They forward the captured NEGOTIATE/CHALLENGE/AUTHENTICATE messages to a target server, authenticating as the victim. Common targets: SMB (file share access, admin operations), LDAP/LDAPS (write to AD attributes — RBCD, addUserToGroup), HTTP (OWA, ADFS), AD CS web enrollment endpoints (ESC8 — request a certificate as the victim, then use it for Kerberos PKINIT). Privileged victims (Domain Admin, computer accounts) yield direct domain compromise.

Mitigations
  • Enforce SMB Signing on all hosts (RequireSecuritySignature=1) — relay across SMB fails
  • Enforce LDAP Signing and LDAP Channel Binding on DCs — closes the LDAP/LDAPS relay path
  • Enable EPA on AD CS web endpoints, OWA, ECP, and any HTTPS that uses NTLM — closes ESC8
  • Disable LLMNR (GPO) and NetBIOS-over-TCP/IP — eliminates name-resolution poisoning
  • Disable IPv6 if not in use, or enforce DHCPv6 guard / RA guard at the switch
  • Patch coercion vectors (Spooler, EFS-RPC PetitPotam, DFS) and consider disabling Print Spooler on servers
  • Restrict NTLM via GPO (Network security: Restrict NTLM) — audit then enforce
Pass-the-Hash (PtH) T1550.002

NTLM authentication does not require the cleartext password — only the NT hash. An attacker holding a stolen hash can inject it into LSASS (or use a tool that performs NTLM auth directly with the hash) and authenticate to any service that accepts NTLM. Used extensively for lateral movement: PSExec, WMI, WinRM, SMB admin shares (C$, ADMIN$), scheduled tasks. Effective until the password is rotated.

Mitigations
  • LAPS for local Administrator passwords — each host has a unique, rotated password, breaking lateral PtH
  • Credential Guard prevents hash extraction in the first place
  • Restrict local account network logon (Deny access to this computer from the network for local accounts) — UAC remote restrictions also block this
  • Network segmentation; jump boxes (PAWs) for admin operations
  • Disable NTLM where possible; enforce Kerberos-only
Hash Capture (Responder-style) T1187 / T1557.001

The attacker stands up a malicious SMB / HTTP / LDAP server and induces clients to authenticate to it (typically by poisoning name resolution). The captured NTLMv2 challenge-response is offline-crackable — for weak passwords, recovery in seconds to hours. Yields the cleartext password, which is then usable everywhere.

Mitigations
  • Disable LLMNR, NBT-NS, mDNS at the host
  • Strong password policy — long passphrases resist offline cracking
  • Block outbound SMB to the internet (egress filtering on TCP/445, /139)
  • Detection: alert on UDP 5355 (LLMNR) traffic; honeypot SMB shares with names like AAA
NTLM Downgrade T1562 / T1556

The attacker manipulates the NTLM negotiate flags or relies on lax server compatibility settings to force NTLMv1, which uses DES on the challenge with the NT hash. NTLMv1 challenge-response is far easier to crack offline (constant 1122334455667788 challenge with crack.sh yields the hash directly).

Mitigations
  • Set LmCompatibilityLevel = 5 via GPO — refuse LM & NTLMv1, send NTLMv2 only
  • Audit 4624 events for NTLMv1 authentications
  • Disable NTLM entirely on servers that don't require it (Restrict NTLM in GPO)

17 — Kerberos Attacks

KRB-OFF
Kerberos overview Event 4769

Kerberos is structurally stronger than NTLM but exposes its own attack surface: ticket forgery (when a long-term key is compromised), offline cracking of tickets encrypted with a service principal's key, abuse of misconfigured delegation, and certificate-based authentication bypasses via AD CS.

Kerberoasting T1558.003

Any authenticated domain user can request a service ticket for any SPN. The TGS-REP contains a portion encrypted with the target service account's long-term key. By preferring RC4-HMAC etype (which uses the NT hash directly as key), the attacker obtains material that can be cracked offline to recover the service account password. Targets accounts with SPNs registered — typically service accounts, often privileged.

Mitigations
  • Use Group Managed Service Accounts (gMSA) — auto-rotated 240-byte random passwords resist offline cracking
  • Disable RC4 on the KDC (Network Security: Configure encryption types allowed for Kerberos) — forces AES, slower to crack
  • Audit accounts with SPNs and ensure long, random passwords
  • Detect: 4769 with Ticket Encryption Type 0x17 (RC4) at high volume from a single source
  • Add sensitive service accounts to Protected Users where compatible
AS-REP Roasting T1558.004

If an account has the DONT_REQ_PREAUTH flag set, the KDC issues an AS-REP without first verifying the client knows the password. The AS-REP contains material encrypted with the user's long-term key — directly offline-crackable. No prior authentication to the domain is required (an unauthenticated attacker with network access to the DC can enumerate and roast).

Mitigations
  • Audit AD for accounts with DONT_REQ_PREAUTH set; remove the flag unless strictly required
  • Detect: 4768 with pre-auth type 0 (no pre-auth) — anomalous when not from known legacy clients
  • Strong password policy — offline crack resistance
Pass-the-Ticket T1550.003

An attacker with admin/SYSTEM on a host extracts Kerberos tickets (TGT or service ticket) from LSASS memory or from credential cache files. The stolen ticket is injected into the attacker's logon session and used as if it were their own. Distinct from PtH because it does not require the user's hash — only the (time-limited) ticket — and the resulting authentication is indistinguishable from legitimate Kerberos to the receiving service.

Mitigations
  • Credential Guard — tickets are isolated in VSM, not extractable from userland
  • Reduce TGT lifetime for sensitive accounts (Protected Users default: 4 hours, non-renewable)
  • Privileged account isolation — admins use PAWs and never log into Tier-1/2 hosts
Golden Ticket T1558.001

Once the krbtgt account's NT hash (RC4) or AES key is compromised, the attacker can forge arbitrary TGTs offline. The forged TGT can claim any user identity, any group memberships (Domain Admins, Enterprise Admins), and any validity period. Persists across user password resets and domain admin removal — only rotating the krbtgt password (twice, due to password history) invalidates them.

Mitigations
  • If krbtgt compromise suspected: rotate the krbtgt password twice (not once — replication and password history matter)
  • Detect: 4769 events for service tickets where the source TGT was never observed at the AS layer (PAC validation logs)
  • Detect: anomalous TGT lifetimes (10+ year tickets are common forgery indicators)
  • Limit Domain Admin / Enterprise Admin membership to absolute minimum; segregate Tier-0
Silver Ticket T1558.002

The attacker compromises a single service account's hash and forges service tickets for that one service. Because the service ticket is only validated by the target service (not the KDC, in standard configurations without PAC validation), no DC interaction is required at use-time — the attack is silent against centralized monitoring. Limited to that specific service.

Mitigations
  • Enable PAC validation (ValidateKdcPacSignature) on services where supported — service contacts DC to verify PAC signature
  • Use gMSA for service accounts
  • Detect tickets without PAC, or with malformed PAC structure (Sysmon and EDR Kerberos parsing)
Overpass-the-Hash T1550.002 + T1558

An attacker holding a user's NT hash (or AES key) sends an AS-REQ with RC4 (or AES) pre-authentication using that key. The KDC issues a legitimate TGT — converting the hash into a Kerberos credential, which is more useful than the raw hash because it works against Kerberos-only services and produces normal-looking authentication telemetry.

Mitigations
  • Same as PtH: Credential Guard, LAPS, Protected Users, network segmentation
  • Disable RC4 — forces AES-only, narrower attack window
  • Detect: 4768 with RC4 etype for accounts that should authenticate AES-only
Unconstrained Delegation Abuse T1558 / T1134

A server with unconstrained delegation receives users' TGTs cached in LSASS. An attacker who compromises that server can extract any TGT that arrives. Combined with a coercion technique (PrinterBug, PetitPotam) to force a Domain Controller's computer account to authenticate to the unconstrained-delegation server, the attacker captures a DC's TGT — sufficient for DCSync and full domain compromise.

Mitigations
  • Audit AD for TRUSTED_FOR_DELEGATION on accounts; minimize use; prefer constrained or RBCD
  • Mark sensitive accounts (DA/EA, DCs) as "Account is sensitive and cannot be delegated" or add to Protected Users
  • Patch coercion vectors and disable Print Spooler on DCs
  • Detect: 4769 for DC computer accounts targeting non-DC SPNs (unusual)
Constrained Delegation / RBCD Abuse T1134.003

Constrained Delegation (KCD): if the attacker compromises a service account configured for KCD, they can use S4U2Self + S4U2Proxy to obtain a service ticket as any user to any SPN listed in msDS-AllowedToDelegateTo. With "use any auth protocol" (protocol transition) flag, no pre-existing user TGT is needed. Resource-Based Constrained Delegation (RBCD): if the attacker can write msDS-AllowedToActOnBehalfOfOtherIdentity on a target computer object (commonly via ACL abuse — see Section 18), they configure the target to trust an attacker-controlled computer account, then S4U2Proxy as Domain Admin to that target and execute as DA. RBCD is the most common ACL-abuse-to-RCE primitive in modern AD.

Mitigations
  • Audit AD for msDS-AllowedToDelegateTo populations and review necessity
  • Audit msDS-AllowedToActOnBehalfOfOtherIdentity across all computers — should normally be empty
  • Restrict who can write to computer objects (ACL hygiene; remove "Authenticated Users" creation rights — MS-DS-Machine-Account-Quota)
  • Add sensitive accounts to Protected Users (cannot be delegated)
  • Detect: 4624 with logon process Kerberos and authentication package S4U; cross-reference with TGS-REQ S4U flags
AD CS Abuse (ESC1–ESC15) T1649

Active Directory Certificate Services trust relationship to Kerberos (PKINIT) and weak default templates / endpoint configurations create a wide range of escalation paths catalogued as ESC1 through ESC15+. Common patterns: ESC1 — request a certificate with arbitrary SAN as a low-privileged user from a misconfigured template (Enrollee Supplies Subject), then PKINIT as Domain Admin; ESC8 — NTLM-relay HTTP enrollment endpoint to issue a certificate as a privileged victim; ESC4 — write access to a certificate template lets you reconfigure it to be vulnerable; Shadow Credentials — write msDS-KeyCredentialLink on a target to add an attacker-controlled key, then PKINIT as the target.

Mitigations
  • Audit certificate templates for ESC1 conditions (Enrollee Supplies Subject + Client Auth EKU + low-privileged enrollment)
  • Disable HTTP enrollment endpoints; if HTTPS, enforce EPA + LDAP signing
  • Restrict template ACLs to admins; review enrollment permissions
  • Audit msDS-KeyCredentialLink writes (Shadow Credentials primitive)
  • Use Microsoft's Certify / Locksmith / PSPKIAudit tools to detect misconfigurations

AdminSDHolder Backdoor T1098

The AdminSDHolder object's DACL is propagated by SDProp every 60 minutes to all members of protected groups (Domain Admins, Enterprise Admins, etc.). An attacker who plants a backdoor ACE (e.g., GenericAll for a controlled account) on AdminSDHolder achieves persistent access — even if defenders manually fix individual user ACLs, SDProp re-applies the malicious template at the next cycle.

Mitigations
  • Audit AdminSDHolder ACL routinely and on every 4670/4662 event affecting it
  • Detect SDProp execution (events 4780) and verify expected ACEs only
  • Monitor any ACE addition to AdminSDHolder as a high-severity alert
SYSVOL / cPassword Exposure T1552.006

Group Policy Preferences (GPP) historically allowed setting passwords for local accounts, scheduled tasks, and services via XML files in SYSVOL. The passwords were AES-encrypted with a key Microsoft published — trivially decryptable (CVE-2014-1812). Patched by removing the password-setting capability, but legacy GPPs still in SYSVOL retain the encrypted blobs and remain readable by all domain users.

Mitigations
  • Audit SYSVOL for cpassword= in *.xml files; remove and rotate any exposed credentials
  • Use LAPS instead of GPP for local account password management
  • Restrict SYSVOL write access to domain admins and Group Policy creators
GPO Abuse — Logon Script / Scheduled Task Injection T1484.001

An attacker with write access to a GPO (or to the SYSVOL files backing it) can inject malicious logon scripts, immediate scheduled tasks, or registry modifications that are applied to every machine and user in scope at next gpupdate. Common edges: GenericWrite on a GPO; write access to a linked OU's gPLink attribute; or membership in Group Policy Creator Owners with edit access to a GPO linked to high-value OUs (like Domain Controllers).

Mitigations
  • Audit GPO ACLs and linkage; least-privilege on Group Policy editing
  • Monitor 4739 (Domain Policy changed), 5136 (Directory object modified) on GPO containers
  • SYSVOL write access tightly controlled; FS auditing on \\domain\sysvol\Policies
  • WDAC / AppLocker on endpoints to block unauthorized scripts
SID History Injection (Cross-Domain) T1134.005

The sIDHistory attribute lets a migrated account retain SIDs from its previous domain. Within a forest, SID history is honored — meaning an attacker who controls a child domain (compromise of any child Domain Admin) can inject SIDs of forest-root Enterprise Admins into a sIDHistory-permitted ticket and escalate forest-wide. The SID Filter Quarantining setting on a trust normally strips foreign SIDs but is disabled within a single forest by design.

Mitigations
  • Treat the forest as the security boundary — child-domain compromise = forest compromise
  • Enable SID Filtering on cross-forest trusts; quarantine external trusts
  • Monitor 4662 events for sIDHistory writes; should be rare outside migrations
  • Detect Kerberos tickets with extra SIDs in PAC that don't match the issuing domain
DnsAdmins → SYSTEM on DC T1574 (legacy)

Members of the DnsAdmins group could historically register a malicious DLL as a DNS plugin via dnscmd /config /serverlevelplugindll; the DNS service (running as SYSTEM on a DC) loads the DLL on next restart, yielding SYSTEM on the DC and effectively domain compromise. Microsoft mitigated by limiting the registry path and patching dnscmd (see CVE-2021-40469 / KB5006744 era), but legacy systems remain vulnerable.

Mitigations
  • Audit and minimize DnsAdmins membership — treat as Tier-0
  • Apply current patches; verify HKLM\SYSTEM\...\DNS\Parameters\ServerLevelPluginDll is restricted
  • Run DNS on dedicated, non-DC servers where feasible

Reference Rail
Kernel Access Path
Object Manager asks SRM to decide whether a token may open a secured object.
Token Ingredients
Authorization is mostly token math: identity, memberships, privileges, and integrity level.
Token Lifecycle
The token is born from logon policy, attaches to a process or thread, then becomes input to SRM.
Token Relationships
Primary, impersonation, UAC, restricted, AppContainer, integrity, and network state are dimensions of one token model.
Access Check Order
A matching deny ACE can end the decision before allow entries finish accumulating rights.
Full Access Pipeline
The DACL is only one phase: generic mapping, MIC, privilege, sandbox, CAP, and audit layers also matter.
NTLM Challenge
The server proves the response by asking the DC, which is why relay defenses matter.
Kerberos Tickets
Kerberos separates identity proof from service access using TGTs and service tickets.
AD Control Path
Small delegated rights can become a control path when they touch high-value objects.
Descriptor Layout
Descriptors bind ownership, discretionary access, and audit intent to each secured object.
LSA Logon Path
Interactive logon asks LSASS to authenticate, build a logon session, and return a token.
URA Enforcement Flow
URA gates logon before a token exists, then token privileges influence later object access decisions.
DPAPI Scope
DPAPI wraps secrets to a user or machine context; audits should inventory, not expose, protected blobs.
Audit Decision Flow
A SACL matters only when audit policy enables that category and logs are collected.
Telemetry Pipeline
The strongest detections join host, DC, directory-change, and network evidence.
